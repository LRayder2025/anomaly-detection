Event flow and reliability: Trace the path of a single CSV file from the moment it is uploaded to raw/ in S3 until the FastAPI app processes it. What happens if the EC2 instance is down or the /notify endpoint returns an error? How does SNS behave (e.g., retries, dead-letter behavior), and what would you change if this needed to be production-grade?
    When test_producer.py uploads a CSV file, it is sent to the raw/ folder in the bucket, and S3 publishes a message to the SNS topic. SNS sees my HTTP endpoint in its subscriptions and sends a POST request with the S3 bucket name and file name. The API reads the names then downloads the file from S3. It then sends the CSV to processor where its statistics are processed and calculated, and then these are merged to baseline.json by way of the baseline.py file. If the Ec2 instance is down, the messages are not lost and SNS will keep retrying to send them, but with longer times between sends as it repeatedly fails. To make this production-grade, we could add a dead-letter queue where failed messages go, or we could use SQS so that they sit in the queue.

Architecture and scaling: This solution uses batch-file events (S3 + SNS) to drive processing, with a rolling statistical baseline in memory and in S3. How would the design change if you needed to handle 100x more CSV files per hour, or if multiple EC2 instances were processing files from the same bucket? Address consistency of the shared baseline.json, concurrent processing, and any tradeoffs.
    For scaling, we could use SQS instead of SNS. This would solve the problem of handling 100x more CSV files per hour because instead of being overloaded with 100 files per minute, each file would be placed in a queue and loaded when it's ready. To address the prolems with multiple EC2 instances processing files from the same bucket, which would result in one instance's data being written over by the other, we could use a distributed database like DynamoDB. This kind of system would receive the output from both instances and merge them, rather than allowing one to overwrite another's baseline.json file. The major tradeoff with this scaled system is complexity. The number of resources and size of systems is growing, and this can lead to further problems, and importantly, monetary cost.
